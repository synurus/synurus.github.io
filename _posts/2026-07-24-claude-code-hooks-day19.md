웹개발 수업 19일차. 어제 스킬(19번)까지 봤던 Claude Code 로드맵을 **20번 훅부터 끝까지** 마무리했다. 그리고 심화로 두 가지를 했다 — **직접 훅을 만들어 설치**했고, 다음 단계인 **플러그인 기획서**를 작성했다. 어제 "내 도구를 내가 만든다"고 썼는데, 하루 만에 도구가 스스로 움직이기 시작했다.

## 1. 수업 내용 요약 — Claude Code 로드맵 (20~23)

### 1-1. 훅 (Hooks) — 이벤트가 터지면 자동 실행

**자동 에어백 비유**가 명쾌했다. 사고(이벤트) 순간 즉시 터지고, 사람이 버튼을 누를 필요가 없다.

- 🚗 에어백 = 훅 스크립트
- 💥 사고 감지 = 이벤트 발생 (파일 저장, 도구 실행 등)
- ⚡ 즉시 팽창 = 설정된 명령어 자동 실행

**이벤트 5종:**

| 이벤트 | 실행 시점 |
|--------|----------|
| `UserPromptSubmit` | 사용자 프롬프트 제출 직후 |
| `PreToolUse` | 도구 실행 **직전** |
| `PostToolUse` | 도구 실행 완료 **직후** |
| `Stop` | Claude 응답 완료 시 |
| `Notification` | Claude 알림 발생 시 |

설정은 `.claude/settings.json`에:

```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Write|Edit",
      "hooks": [{ "type": "command", "command": "node .claude/html-check.js" }]
    }]
  }
}
```

`matcher`로 **어떤 도구일 때** 발동할지 거르고, `command`로 실행할 명령을 지정한다. 12일차 Express의 미들웨어(모든 요청이 라우트 도착 전에 거쳐가는 검색대)와 구조가 똑같다 — **본 작업 앞뒤에 끼어드는 공통 처리**.

### 1-2. 스킬 vs 훅 — 배달 앱 vs 자동문 센서

| | 📱 스킬 | 🚪 훅 |
|---|--------|-------|
| 비유 | 배달 앱 | 자동문 센서 |
| 발동 | **내가 부를 때만** (`/명령어`) | **조건이 되면 저절로** |
| 성격 | 평소엔 대기 중 | 내가 아무것도 안 해도 감시·처리 |
| 예 | `/code-review` | 파일 수정 후 자동 문법 검사 |

어제 만든 markdown-reader 스킬 파일에 "새로 만든 .md는 훅이 이미 처리한다"고 적어뒀는데, 그 역할 분담의 정체가 오늘 정리됐다.

### 1-3. Codex CLI 연동 (6단계)

Claude Code에 OpenAI Codex를 플러그인으로 붙이는 방법:

```
/plugin marketplace add openai/codex-plugin-cc   ① 마켓플레이스 추가
/plugin install codex                            ② 설치
/reload-plugins                                  ③ 재로드
/codex setup                                     ④ 셋업
!codex login                                     ⑤ 로그인
/codex:review                                    ⑥ 리뷰 실행
```

**서로 다른 AI로 교차 리뷰**를 받을 수 있다는 발상이 신선했다. 17일차 심화에서 "막히면 다른 도구로 갈아탄다"를 배웠는데, 여기선 아예 둘을 같이 쓴다.

### 1-4. Claude Cowork — 비서처럼 부리기

Claude App + 브라우저 + 외부 서비스(카카오·네이버 등)를 연결해 복잡한 작업을 자동 처리하는 환경.

**5단계 설정:** ① 작업 폴더 준비 → ② Claude 앱 설치 + 폴더 권한 부여 → ③ Chrome 확장 연결 → ④ **PlayMCP 추가**(카카오톡·네이버 검색·날씨 사용) → ⑤ 활용

활용 예: *"오늘 뉴스 요약해줘"*, *"한국 증시 분석해"*, *"Word 문서로 저장해"*, *"카카오톡으로 보내줘"*. 그리고 **매일 자동 실행 스케줄**까지 걸 수 있다 (예: 매일 오전 8시 증시 분석 + 카톡 전송).

17일차에 배운 MCP(도구 연결의 USB-C 표준)가 여기서 실제 제품으로 등장했다 — 개념 → 구현 → 활용의 순서로 사흘에 걸쳐 이어진 셈.

## 2. 어려웠던 것 — 심화 학습

### 2-1. 훅 제작 — "바퀴를 다시 만들지 마" 훅 🔁

**문제의식:** Claude Code로 뭔가 기능을 만들 때, **이미 설치해둔 스킬로 해결되는 일을 처음부터 새로 코딩**하는 경우가 있다. 스킬이 늘어날수록 내가 뭘 깔았는지도 잊어버린다. 어제 스킬을 만들면서 생긴 새 문제였다.

**해결:** 파일을 쓰기 **직전**에, 설치된 스킬 목록을 Claude에게 슬쩍 알려주는 훅.

**⑴ 어떤 이벤트를 골랐나 — PreToolUse**

```
도구: Write, Edit
시점: 실제 파일 작성 직전 (도구 호출은 허용, 차단 아님)
```

`PostToolUse`(사후)는 이미 파일을 쓴 뒤라 늦고, `UserPromptSubmit`(요청 시점)은 아직 뭘 만들지 정해지기 전이다. **"쓰기 직전"이 개입의 최적 타이밍**이라 `PreToolUse`를 골랐다.

**⑵ 스크립트 — 폴더를 스캔해 목록을 만든다**

```python
# check-skills.py (PowerShell 버전 check-skills.ps1도 동일 동작)
skills_dir = Path.home() / '.claude' / 'skills'
installed_skills = sorted([d.name for d in skills_dir.iterdir() if d.is_dir()])
skills_list = ", ".join(installed_skills) if installed_skills else "(없음)"

context = f"""설치된 스킬: {skills_list}

지금 만들려는 기능이 이 중 하나로 이미 커버되면 새로 작성 말고 그 스킬 사용을
사용자에게 알릴 것. 커버 안 되면 /find-skills 스킬로 skills.sh 검색해서 제안할 것."""

output = {
    "hookSpecificOutput": {
        "hookEventName": "PreToolUse",
        "permissionDecision": "allow",      # 차단이 아니라 통과
        "additionalContext": context        # 여기에 담은 글이 Claude에게 주입됨
    }
}
json.dump(output, sys.stdout, ensure_ascii=False, separators=(',', ':'))
```

핵심은 **훅이 stdout에 뱉은 JSON이 곧 Claude의 컨텍스트가 된다**는 것. `additionalContext`에 담은 문장이 그대로 "지금 이 순간 Claude가 읽는 지침"이 된다. 12일차에 배운 **system prompt를 상황에 맞춰 동적으로 주입**하는 셈이다.

`"permissionDecision": "allow"`도 중요한 선택 — 훅은 도구 실행을 **막을 수도 있는데**, 여기선 막지 않고 통과시키면서 정보만 얹었다.

**⑶ 설계에서 신경 쓴 두 가지**

**판단 주체를 분리했다.**

| | 역할 |
|---|------|
| 훅 | 목록만 **주입** (강제하지 않음) |
| Claude 모델 | 실제 관련성 **판단** 및 실행 결정 |

훅이 "무조건 스킬 써!"라고 강제하면 오히려 방해가 된다. **판단은 모델에게 맡기고, 훅은 판단할 재료만 준다** — 이 경계가 훅 설계의 핵심이었다.

**스킬 인식 기준은 폴더 존재 여부로.** 처음엔 `.skill-lock.json`(설치 기록 파일)을 읽으려 했는데, **수동 설치나 번들로 들어온 스킬이 lock 파일에 등록되지 않는** 경우가 있었다. 그래서 `.claude/skills/<이름>` **디렉토리가 있으면 설치된 것**으로 판단하도록 바꿨다. 메타데이터보다 실제 상태를 믿는 쪽.

**⑷ 파이썬·PowerShell 두 벌로 만든 이유**

같은 동작을 `check-skills.py`와 `check-skills.ps1` 두 버전으로 만들었다. 인코딩 처리가 양쪽 다 첫 줄에 들어간다:

```python
sys.stdout.reconfigure(encoding='utf-8')          # Python
```
```powershell
[Console]::OutputEncoding = [System.Text.Encoding]::UTF8   # PowerShell
```

한글이 섞인 컨텍스트를 주입하니 **UTF-8 명시가 필수**였다. 14일차 파일 입출력에서 `encoding="utf-8"`을 배웠던 게 여기서 또 나온다 — 한글은 어디서든 인코딩을 챙겨야 한다.

**⑸ 지금 돌고 있는 훅 4종**

이번 것까지 합쳐 훅이 네 개가 됐다:

| 이벤트 | 하는 일 |
|--------|---------|
| `PreToolUse` | **설치된 스킬 목록 주입** (오늘 만든 것) |
| `PostToolUse` | Write 후 `.md` 자동으로 열기 |
| `UserPromptSubmit` | `.md` 파일 경로만 입력하면 열기 |
| `Stop` | 응답 완료 후 알림음 + 메시지 |

어제 SKILL.md에 적어둔 *"새로 만든 .md와 경로만 드래그한 경우는 훅이 처리한다"*의 실체가 2·3번이다. **스킬(명시적 요청) + 훅(자동)** 조합이 실제로 굴러가는 걸 확인했다.

### 2-2. 플러그인 기획 — 라이브 방송 인기 카테고리 분석 📊

스킬 → 훅 다음은 **플러그인**. 아직 제작 중이라 오늘은 기획서까지 썼다.

**만들려는 것:** 트위치·치지직의 **인기 카테고리(게임/콘텐츠 장르)를 자동 수집·분석**해 정기적으로 마크다운 리포트를 받아보는 플러그인. 스트리밍 트렌드는 신작 출시·대회 시즌 등에 따라 카테고리 단위로 빠르게 움직이는데, 이걸 수작업으로 추적하기가 번거로워서.

**① 실현 가능성부터 따졌다** — 기획서에서 제일 오래 걸린 부분:

| 플랫폼 | 공식 API | 인증 | 판정 |
|--------|---------|------|------|
| **트위치** | Helix (`Get Streams`, `Get Top Games`) | Client ID + App Token | ✅ 바로 가능 |
| **치지직** | 오픈 API (`GET /open/v1/lives`) | Client ID + Secret | ✅ 가능 |
| **숲(SOOP)** | LIVE 오픈 API | API Key (**파트너십 심사 ~10영업일**) | ⚠️ 2차 확장 |

승인 심사가 필요한 숲은 과감히 2차로 미루고 1차 범위를 확정했다. **"할 수 있는 것부터 자르기"** — 11일차 K-POP 차트 앱에서 배운 교훈(공개 API는 원하는 모양대로 안 준다)의 연장선이다.

실제로 데이터 함정도 하나 찾았다 — 트위치의 `Get Top Games`는 카테고리 목록은 주지만 **시청자 수를 안 준다.** 그래서 `Get Streams`를 시청자 내림차순으로 여러 페이지 받아 `game_id` 기준으로 **직접 합산**하는 방식으로 설계했다.

**② 구조 — 오늘까지 배운 게 다 들어간다**

```
[스케줄 작업] --주기적 실행--> [스킬]
                                 ↓
                       [수집·분석 스크립트]
                        /              \
             [트위치 Helix]        [치지직 오픈 API]
                        \              /
                    [공통 스키마 집계 데이터]
                                 ↓
                       [마크다운 리포트 생성]
```

- **수집·분석 스크립트(Python)** ← 17일차 OpenAPI + pandas 집계
- **스킬(SKILL.md + 스크립트)** ← 18일차 스킬 제작
- **스케줄 작업(정기 실행)** ← 오늘 배운 자동 실행

**③ 미리 적어둔 리스크 4가지**

기획서에서 가장 중요한 부분은 "될 것 같다"가 아니라 **"뭐가 문제될까"**였다:

| 리스크 | 내용 |
|--------|------|
| **API 키 저장** | 정기 작업은 매번 **새 세션**에서 시작 → 키를 어딘가 보관해야 함. 프롬프트 내장(간단) vs 연결 폴더의 설정 파일(권장) |
| **트렌드 추적** | 세션 파일시스템은 매 실행 시 **초기화** → 회차 비교하려면 스냅샷 CSV를 연결 폴더에 누적 저장 |
| **레이트 리밋** | 수집 페이지 수를 합리적으로 제한 |
| **카테고리 매핑** | 트위치는 게임 중심, 치지직은 GAME/SPORTS/ETC 유형 구분 → **1:1 매핑 불가**, 명칭 정규화 규칙 별도 관리 |

특히 **"새 세션마다 초기화된다"**는 제약이 인상적이었다. 브라우저 localStorage(7일차), 서버의 `.env`(12일차)에 이어 **"상태를 어디에 남길 것인가"** 문제를 또 만난 것. 실행 환경이 바뀌어도 질문은 똑같다.

**④ 1차는 스냅샷으로 시작**

트렌드 비교는 히스토리가 쌓여야 가능하니, **1차 버전은 "그 시점 스냅샷"만** 만들고 누적은 나중에 붙이기로 했다. 8일차 미니 프로젝트에서 배운 *"작게 시작해서 기능 하나씩"*을 기획 단계에서 적용한 셈.
